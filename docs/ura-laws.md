# URA Laws

## Law 1 — Everything is a canvas

No fixed screens. Components receive data and render only what is meaningful.

## Law 2 — Context is the primary driver

The app owns context. The library exposes contracts.

## Law 3 — Time is the core optimization metric

Fewer useful steps beat prettier longer flows.

## Law 4 — Ideal interface equals zero interface

Hide sections with no meaningful data.

## Law 5 — One input, many outcomes

One user intent can trigger multiple coordinated app behaviors.

## Rule 6 — Universal interface stack

Canvas → Setting → Context → Source → View → Interaction.

## Rule 7 — Source-view decoupling

Views never know where data came from.

## Rule 8 — Data shape determines view

Data shape selects the component pattern.

## Rule 9 — Max data is not view data

Render the focused subset.

## Rule 10 — Views define interaction limits

No action exists unless it can be executed cleanly.

## Rule 11 — Separate input from discovery

Input and discovery are different surfaces.

## Rule 12 — Search is mandatory for overflow

Select > 10, list > 20, table > 50.

## Rule 13 — Defaults beat configuration

Smart defaults must be overridable.

## Rule 14 — Predict before the user acts

Surface likely actions first.

## Rule 15 — F.O.R.D. sorting

Sort by frequency, occurrence, recency, and depth.

## Rule 16 — Split viewport intelligently

Predicted/high-probability content first; full dataset second.

## Rule 17 — Recursion is a law

Components must be nestable.

## Rule 18 — Views are reusable across contexts

Use generic views, never business-specific views.

## Rule 19 — Pattern engine over pattern library

Let data shape, context, and capability determine rendering.

## Rule 20 — Constraints are features

Device, bandwidth, role, permissions, and modality reduce noise.

## Rule 21 — Device-aware by props

Overlay accepts `mode: "modal" | "sheet" | "drawer"`.

## Rule 22 — Extensible without rewrite

New views, roles, and modalities plug in through metadata.
